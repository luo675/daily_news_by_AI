# 架构设计

## 1. 设计原则

- 先做高质量知识沉淀，再做复杂自动化
- 先做个人可用系统，再预留可扩展能力
- 优先输出结构化结果，而不是堆积文本
- 冲突信息优先自动判断，无法判断时显式保留不确定性
- 所有重要输出尽量可追溯到来源证据

## 2. 总体分层

系统建议拆为五层：

1. 数据采集层
2. 数据处理层
3. 知识存储层
4. 检索与判断层
5. 应用与 API 层

### 2.1 数据采集层

负责从外部来源获取原始内容或文字稿。

来源类型：

- 技术博客
- 演讲文字稿
- 访谈文字稿
- 播客文字稿
- 手工导入内容

输出：

- 统一格式的原始文档对象
- 基础来源元数据

### 2.2 数据处理层

负责把原始内容加工为可用知识。

处理能力：

- 清洗与标准化
- 去重
- chunk 切分
- 摘要生成
- 标签与关键词抽取
- 实体抽取
- 主题归类
- 冲突检测
- 来源可信度评分
- 产品机会信号抽取

输出：

- 结构化摘要
- 可检索 chunk
- 实体与主题结果
- 冲突与不确定性标记
- 产品机会候选项

### 2.3 知识存储层

负责保存加工后的结构化资产。

存储内容：

- 来源数据
- 文档与摘要
- chunk 与 embedding
- 实体、主题、事件、关系
- 日报结果
- 机会评分结果
- 人工修订记录

第一阶段不强求保存全文或网页快照，重点保存摘要、结构化结果和来源链接。

### 2.4 检索与判断层

负责把知识转成可消费输出。

能力包括：

- 关键词检索
- 语义检索
- 混合检索
- 重排序
- watchlist 加权
- 冲突判断
- 产品机会评分
- 趋势与风险归纳

### 2.5 应用与 API 层

面向用户和通用 AI Agent 提供能力。

包含：

- 每日简报
- 专题聚合
- 产品机会输出
- 人工修订入口
- 面向 AI Agent 的结构化 API

## 3. 模块划分

### 3.1 `source_registry`

职责：

- 管理来源类型
- 管理来源可信度等级
- 管理来源启停状态
- 管理采集策略

### 3.2 `document_ingestion`

职责：

- 拉取外部内容
- 统一文档输入格式
- 记录文档基础元数据

### 3.3 `document_processing`

职责：

- 清洗
- 去重
- 分段
- 摘要
- 标签、实体、主题抽取

### 3.4 `knowledge_indexing`

职责：

- 结构化入库
- 构建向量索引
- 建立检索索引

### 3.5 `conflict_resolution`

职责：

- 检测事实冲突、观点冲突、时间冲突
- 结合来源可信度做倾向判断
- 标记不确定性

### 3.6 `opportunity_scoring`

职责：

- 抽取产品机会候选项
- 生成评分
- 关联证据
- 输出结构化判断结果

### 3.7 `daily_briefing`

职责：

- 聚合新增内容
- 生成中英双语日报
- 支持固定日报与按需简报

### 3.8 `watchlist`

职责：

- 管理关注对象
- 管理优先级与分组
- 为检索与日报提供加权上下文

### 3.9 `agent_api`

职责：

- 提供搜索、日报、机会、主题等接口
- 保证结构化返回
- 管理鉴权与配额

### 3.10 `admin_review`

职责：

- 允许人工修订摘要、标签、评分、结论
- 覆盖自动处理结果
- 保留审计记录

## 5. 核心数据流

建议的数据流：

1. 来源注册
2. 文档采集
3. 元数据标准化
4. 清洗与去重
5. 摘要与抽取
6. 入库与索引
7. 冲突检测
8. 机会评分
9. 日报生成
10. API 输出
11. 人工修订回写

## 6. 存储架构

建议第一阶段采用：

- `PostgreSQL`：主数据、结构化结果、审计记录
- `pgvector`：chunk 向量索引
- 文件系统目录：中间结果、日报导出、调试输出

优点：

- 架构简单
- 易于单机部署
- 足够支撑第一阶段规模

## 7. 关注列表设计

关注列表支持两种组织方式：

- 按对象类型
- 按优先级

支持对象：

- 人物
- 公司
- 产品
- 模型
- 主题
- 赛道
- 关键词

关注列表在以下场景参与决策：

- 检索重排序
- 日报优先级提升
- 专题聚合
- 机会判断增强

## 8. 冲突处理设计

冲突类型至少包括：

- 事实冲突
- 观点冲突
- 时间冲突

判断原则：

- 一手来源优先
- 高可信来源优先
- 无法明确判断时保留不确定性
- 最终输出显式标记存在分歧

## 9. 人工修订设计

人工修订支持：

- 摘要
- 标签
- 主题
- 机会分数
- 风险标记
- 结论与优先级

原则：

- 人工结果优先级高于自动结果
- 所有改动保留版本记录

## 10. 技术选型建议

- 后端：`Python + FastAPI`
- 数据库：`PostgreSQL`
- 向量：`pgvector`
- 调度：`Celery` 或 `Prefect`
- 抓取：`RSS + requests`，必要时 `Playwright`
- 模型接入：可插拔接口，优先低成本中国模型
- 部署：`Docker Compose`

## 11. 风险与扩展方向

主要风险：

- 来源质量不稳定
- 自动摘要失真
- 冲突判断过度简化
- 机会评分过度主观
- API 结构不稳定影响 Agent 可用性

后续扩展方向：

- 音视频转录接入
- 更复杂的来源调度
- 更强的图谱分析
- 多用户权限与分享机制
