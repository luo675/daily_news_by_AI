# 项目目标与架构初稿

## 1. 项目愿景

最终目标是建立一个独属于自己的 AI 领域知识数据库。

这个数据库不仅保存信息，还要能持续吸收新的 AI 资讯、理解历史内容、建立内容之间的关联，并为后续的检索、问答、趋势判断、每日简报生成提供基础能力。

项目希望达到的效果是：

- 收集网上各种 AI 大佬的演讲、对话、访谈、文章、新闻、资料
- 对这些内容进行清洗、处理、分类、摘要、关联
- 用结构化方式保存处理后的数据
- 让通用 AI 在读取这些数据后，能把这些领域知识纳入判断过程
- 即使没有最强通用模型，也能在特定问题上给出更好的判断
- 在历史知识库的基础上，持续收集每天最新资讯，自动生成每日简报，并和过去的信息串联起来

## 2. 项目要解决的问题

当前 AI 领域的信息分散在新闻网站、博客、播客、视频、社交平台、采访和公开演讲中，存在以下问题：

- 信息来源分散，难以系统整理
- 内容更新很快，靠人工跟踪成本高
- 高价值内容容易被埋在大量噪声中
- 同一主题跨时间演化，人工很难持续串联
- 通用 AI 缺少专门整理过的领域知识上下文，导致判断泛化但不够深

项目的核心目标是解决以下问题：

1. 把高价值 AI 领域信息沉淀成可持续积累的私有知识库
2. 让信息从“原始资料”变成“可检索、可追溯、可关联、可复用”的知识资产
3. 支持按天追踪新动态，并自动形成每日简报
4. 支持围绕主题、人物、公司、模型、事件进行长期跟踪和分析
5. 支持投资研究和产品机会判断

## 3. 核心用户

当前默认核心用户：

- 项目拥有者本人
- 第一阶段按个人研究工具设计

潜在扩展用户：

- AI 行业研究者
- 投资分析者
- 技术内容创作者
- 产品经理或战略分析人员
- 需要持续跟踪 AI 领域动态的专业用户

成果传播方式：

- 第一阶段主要供本人使用
- 后续会把整理后的成果分享给其他人

## 4. 主要功能

### 4.1 数据采集

- 收集 AI 领域新闻、文章、博客、演讲、访谈、播客、视频字幕、公开对话等内容
- 支持多种采集方式，如 RSS、API、网页抓取、手工导入、转录导入
- 为每条内容记录来源、发布时间、作者、链接、抓取时间等元数据

第一阶段优先采集的数据类型：

- 技术博客
- 演讲
- 访谈
- 先处理已有文字稿的播客或访谈内容

后续扩展：

- 音视频转文字

### 4.2 数据处理

- 文本清洗和标准化
- 内容去重
- 分段切块
- 摘要生成
- 标签分类
- 关键词抽取
- 实体识别
- 事件和主题提取
- 冲突信息检测
- 来源可信度分级

### 4.3 知识存储

- 保存处理后的结构化数据
- 保存摘要
- 保存向量索引用于语义检索
- 保存实体、事件、主题、关联关系
- 保存原始链接和必要元数据

第一阶段存储策略：

- 不强求保存原始全文或网页快照
- 优先保存摘要、处理结果和后续分析结果

### 4.4 检索与问答

- 关键词检索
- 语义检索
- 混合检索
- 基于证据的问答
- 查询结果可回链到原始来源
- 支持通过 API 向外部 AI 提供知识读取能力

### 4.5 每日简报

- 收集每天新增内容
- 自动去重、聚合、分类
- 生成结构化每日简报
- 把当日信息与过去主题、人物、事件串联

第一阶段输出形式：

- Markdown

生成方式：

- 每天固定生成一份
- 当用户有需求时，可以额外读取最新数据再生成一份新的简报

后续可扩展输出形式：

- 邮件
- 飞书或企业微信

### 4.6 专题串联与判断输出

- 按人物跟踪观点变化
- 按公司跟踪产品和战略变化
- 按模型跟踪能力路线
- 按事件形成时间线和背景脉络
- 面向投资研究输出机会清单、风险清单、趋势判断
- 面向产品机会输出方向列表、产品想法、市场空白

### 4.7 人工修订

- 支持人工修改摘要、标签、结论、优先级和判断结果
- 人工权限高于自动处理结果

## 5. 总体架构

项目建议采用“离线处理 + 在线检索/生成”的分层架构。

### 5.1 架构分层

#### 数据采集层

负责从外部来源收集原始数据：

- RSS
- API
- 网页抓取
- 手工导入
- 文字稿导入
- 后续支持音视频转录导入

#### 数据处理层

负责把原始内容加工为可用知识：

- 清洗
- 标准化
- 去重
- chunk 切分
- 摘要
- 标签
- 实体抽取
- 关系抽取
- 向量化
- 冲突检测
- 可信度评分

#### 知识存储层

负责存储不同层级的数据：

- 结构化数据库
- 向量数据库
- 关联关系存储
- 应用层输出存储

#### 检索与推理层

负责支持上层应用：

- 关键词检索
- 向量检索
- 混合检索
- 重排序
- 上下文组装
- 冲突判断
- 带来源引用的回答生成

#### 应用层

面向用户提供能力：

- 每日简报
- 专题汇总
- 知识问答
- 数据管理后台
- 外部 API 接口，供其他 AI 或工具读取知识库

### 5.2 推荐技术架构

建议优先使用简单、可控、易扩展的方案：

- 后端：Python + FastAPI
- 任务调度：Celery 或 Prefect
- 数据库：PostgreSQL
- 向量检索：pgvector 或 Qdrant
- 抓取：RSS + requests + Playwright 按需补充
- 模型能力：可插拔的 LLM / embedding 接口，优先考虑可接入中国低成本模型
- 管理界面：轻量 Web 后台
- 部署：Docker Compose 起步

第一阶段实现边界：

- 先做个人可用的内部系统
- 同时预留对外 API 能力，支持外部 AI 读取知识库
- 先采用手动触发的数据更新方式
- 系统稳定后再增加自动化调度

### 5.3 核心存储设计

建议至少分成四类存储对象：

1. 文档元数据
   - 来源
   - 发布时间
   - 作者
   - URL
   - 采集时间
   - 语言
   - 来源类型
   - 可信度等级
   - 处理状态

2. 处理结果
   - 摘要
   - 标签
   - chunk
   - embedding
   - 实体
   - 主题
   - 事件
   - 冲突标记

3. 关联数据
   - 人物关系
   - 公司关系
   - 模型关系
   - 事件关系
   - 主题聚合

4. 应用结果
   - 每日简报
   - 专题聚合
   - 判断输出
   - 问答引用记录

## 6. 高风险点

### 6.1 数据版权与合规

- 是否允许保存全文
- 是否允许保存网页快照
- 是否允许转录和再加工公开内容
- 是否允许对外分发处理结果

### 6.2 数据质量

- 来源质量参差不齐
- 重复内容很多
- 自动摘要可能失真
- 实体和主题抽取可能不稳定
- 不同来源对同一事件的描述可能互相矛盾
- 项目不要求所有结论强制逐条回链，但要求最终输出尽量准确，因此需要冲突检测与判断机制

冲突处理原则：

- 优先自动判断
- 如果系统无法形成足够可信的判断，则明确保留不确定性
- 在最终输出中标注存在分歧或待确认的信息

来源可信度要求：

- 对来源做可信度分级
- 区分一手来源、原始发言、机构原文、二手转述、评论类内容
- 在冲突判断时把来源可信度纳入权重

### 6.3 检索质量

- chunk 切分不好会影响召回
- 向量检索容易语义相近但事实不准
- 缺少重排序会导致结果不稳定

### 6.4 自动关联能力

- 人物、公司、模型、事件之间的关系抽取难度高
- 历史串联如果设计不好，很容易变成普通新闻聚合
- 投资研究和产品机会判断如果没有明确模板，输出容易空泛

### 6.5 成本与运维

- 抓取、转录、embedding、摘要、日报生成都有成本
- 定时任务和失败重试如果缺失，系统长期运行会不稳定

## 7. 关键依赖项

- 可持续的数据源清单
- 稳定可调用的模型服务
- PostgreSQL 和向量检索环境
- 定时任务系统
- 数据处理流水线
- 人工校验和反馈机制
- 外部 API 设计与权限控制

## 8. 分阶段计划

### M1. 需求定义与知识模型设计

目标：

- 明确业务目标
- 明确 MVP 范围
- 明确目标用户
- 明确数据源优先级
- 设计核心数据模型

当前已确认的业务方向：

- 使用者：第一阶段为个人使用
- 分享方式：后续分享研究成果给其他人
- 核心价值：服务投资研究和产品机会判断
- 语言策略：英文优先，必要时可在输出阶段翻译为中文

交付：

- 项目定义文档
- 数据源清单
- 核心实体关系草案
- 第一版架构方案

### M2. 采集与入库闭环

目标：

- 打通少量高价值数据源的采集
- 建立元数据和处理结果存储
- 建立处理状态流转

交付：

- 数据采集链路
- 数据库存储规范
- 数据库 schema
- 基础处理流水线

### M3. 检索与每日简报 MVP

目标：

- 建立关键词和语义检索能力
- 支持增量内容汇总
- 自动生成每日简报
- 提供外部 AI 可调用的读取 API 初版

交付：

- 检索接口
- 向量索引
- 每日简报生成流水线
- 日报模板
- 手动触发流程
- 外部 API 初版

### M4. 知识关联与专题能力

目标：

- 建立人物、公司、模型、事件关联
- 支持专题汇总和历史串联
- 支持投资研究和产品机会判断输出

交付：

- 实体关系模型
- 专题聚合能力
- 历史串联逻辑
- 机会清单、风险清单、趋势判断
- 方向列表、产品想法、市场空白

### M5. 运维、质量与长期演进

目标：

- 保证系统可长期运行
- 建立质量评估和反馈闭环
- 控制成本与提高稳定性

交付：

- 监控与告警
- 质量指标
- 人工校验机制
- 成本追踪方案
- 自动化调度能力

补充要求：

- 需要人工修订入口
- 后续应支持人工修改摘要、标签、判断结果和重点结论
- 人工权限最大，可覆盖自动处理结果

## 9. 任务框架

### 9.1 M1 任务

#### 任务 M1-T1：明确业务目标

输入：

- 初始目标描述
- 用户场景说明

输出：

- 清晰的问题定义
- 核心用户定义
- MVP 边界

验收标准：

- 能清楚回答做给谁用、解决什么问题、第一阶段不做什么

#### 任务 M1-T2：确定数据源优先级

输入：

- 候选来源列表

输出：

- P0 / P1 / P2 数据源清单

验收标准：

- 每个 P0 来源都明确采集方式和优先级理由

#### 任务 M1-T3：设计知识模型

输入：

- 样本文档
- 使用场景

输出：

- Source / Document / Chunk / Entity / Topic / Event / Relation / DailyBrief 模型草案

验收标准：

- 数据模型能支撑采集、处理、检索、日报和专题关联

### 9.2 M2 任务

#### 任务 M2-T1：打通数据采集

输入：

- P0 数据源

输出：

- 采集链路
- 原始文档数据或文字稿

验收标准：

- 至少 3 个来源可以稳定采集

#### 任务 M2-T2：建立存储结构

输入：

- 数据模型草案

输出：

- 数据库表结构
- 元数据存储规范

验收标准：

- 元数据、处理结果、状态、判断输出都可落库

#### 任务 M2-T3：建立处理流水线

输入：

- 原始文档或文字稿

输出：

- 清洗结果
- chunk
- 摘要
- 标签
- 实体
- 可信度等级
- 冲突标记

验收标准：

- 文档可从原始态稳定进入可检索态

### 9.3 M3 任务

#### 任务 M3-T1：建立检索能力

输入：

- 已处理文档

输出：

- 关键词检索
- 语义检索
- 混合检索

验收标准：

- 给定主题时可以返回高相关片段和来源

#### 任务 M3-T2：建立日报生成能力

输入：

- 当日新增文档

输出：

- 每日简报

验收标准：

- 日报结构稳定、内容可读、结果可追溯

#### 任务 M3-T3：建立外部读取 API

输入：

- 已处理知识数据

输出：

- 可供外部 AI 调用的读取接口

验收标准：

- 外部系统可按主题、关键词、时间范围读取知识结果

### 9.4 M4 任务

#### 任务 M4-T1：建立知识关联

输入：

- 历史文档
- 实体抽取结果

输出：

- 人物、公司、模型、事件的关联记录

验收标准：

- 可基于人物或主题查看历史脉络

#### 任务 M4-T2：建立专题聚合

输入：

- 多日、多来源内容

输出：

- 专题页面或专题摘要

验收标准：

- 同一主题可以跨时间串联，不只是单日摘要

#### 任务 M4-T3：建立判断输出模板

输入：

- 历史主题数据
- 当日新增内容

输出：

- 投资机会清单
- 风险清单
- 趋势判断
- 产品方向列表
- 产品想法
- 市场空白

验收标准：

- 输出结构固定，可人工修订，不是泛泛总结

### 9.5 M5 任务

#### 任务 M5-T1：建立质量评估体系

输入：

- 查询样本
- 处理结果

输出：

- 评估指标
- 质量报告

验收标准：

- 可以量化检索质量、摘要质量、去重效果、判断准确性

#### 任务 M5-T2：建立运维机制

输入：

- 系统日志
- 任务运行记录

输出：

- 监控指标
- 告警机制
- 成本记录

验收标准：

- 任务失败、数据异常、成本异常可以及时发现

### 产品机会判断默认权重方案

第一阶段建议采用以下默认权重：

- 需求真实性：30%
- 市场空白度：30%
- 产品化可行性：20%
- 跟进优先级：10%
- 证据充分度：10%

设计原则：

- 需求真实性和市场空白度权重最高，因为这是当前最核心的判断标准
- 产品化可行性用于过滤“看起来有趣但做不出来”的机会
- 跟进优先级用于帮助后续行动排序
- 证据充分度用于防止基于少量噪声信息做出过强结论

## 11. 当前工作原则

- 先明确问题，再做实现
- 先做 MVP，再做复杂能力
